# Commit and Push

Stages all changes, creates a conventional commit, and pushes to the current remote branch.

## When to use

When the user asks to commit and push changes — e.g. "commit it", "create a commit and push", "commit and push".

## Steps

1. Run `git status` to see what changed.
2. Run `git diff` to review staged + unstaged changes.
3. Stage the relevant files by name (never `git add -A` blindly — skip `.env`, secrets, unrelated files).
4. Draft a commit message following the project's conventional commit format:
   - `type(scope): short imperative description`
   - Types: `feat`, `fix`, `chore`, `refactor`, `docs`, `style`, `test`, `perf`
   - Keep the subject line under 72 characters
   - **No Co-Authored-By** — never add AI attribution
5. Commit using a HEREDOC to preserve formatting.
6. Run `git push` (or `git push -u origin <branch>` if the branch has no upstream yet).
7. Confirm success with `git status`.

## Conventions

- Separate unrelated changes into individual commits — never bundle them
- Use the Linear issue ID in the commit body when relevant: `[PORT-12]`
- No emojis, no "Generated by", no AI attribution anywhere
- **No vague commit messages** — never write `fix bug`, `update styles`, `misc changes`, `WIP`, or similar. The message must say exactly what changed and why. Bad: `fix seo`. Good: `feat(seo): add alternateName variants to Person schema for name-query indexing`.

## Example

```bash
git add src/layouts/BaseLayout.astro src/components/Hero/Hero.astro
git commit -m "$(cat <<'EOF'
feat(seo): add alternateName variants and fix sameAs links for name indexing
EOF
)"
git push
```
